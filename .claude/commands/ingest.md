请执行 Ingest 操作：摄入指定文件或扫描 `raw/` 下未处理的新文件，将原文**深度编译**为结构化 wiki。

## 输入

- `/ingest <path>` → 摄入该文件。
- `/ingest` → 扫描 `raw/`，找出 `wiki/log.md` 未记录的文件，逐个处理。

## 核心质量原则

Wiki 页面的目标是**迷你百科**，不是索引卡。每个概念页必须能让读者"看完这一页不需要回翻源文"。ingest 失败的典型样貌：一页只有摘要 + 4~5 条 bullet + 关联列表，公式/代码/具体数字都丢了。你必须主动对抗这种压缩倾向。

## 只增不改原则（数据安全红线）

**Ingest 是只增（accretive）操作**。本次 ingest 中你只能：

- 在 `wiki/concepts/`、`wiki/entities/`、`wiki/summaries/` 下**创建新文件**
- 用 `Edit` 工具向 `wiki/_index.md` 和 `wiki/log.md` **追加内容**
- 用 `Edit` 工具修改本次新建的页面，或按 B5 流程**追加**到已有页面

**严禁**：

- 删除任何 `wiki/` 下的已有文件（不论是不是本次创建的）
- 用 `Write` 工具**重写** `wiki/_index.md`、`wiki/log.md` 或任何已存在的页面——这会无声销毁前次 ingest 的成果
- 修改本次 ingest 之外、其他 ingest 创建的页面内容（除非是 B5 合并/矛盾标记，且必须保留原内容）

如果发现 `wiki/_index.md` 或 `wiki/log.md` 已存在内容，正确做法是 `Read` → 用 `Edit` 追加新条目；错误做法是 `Write` 覆盖。两者结果差天差地。

## 阶段 0 — 源文分类（决定后续分支）

每个源文先用 `Read` 读**前 80 行**（或整个文件，如果短），然后按以下规则分类：

| 类型 | 判定信号 | 典型例子 |
|------|----------|----------|
| **dialogue** | 含 `**用户**：` / `**助手**：` 格式标记 | DeepSeek 分享对话、ChatGPT 导出 |
| **structured** | 有多级 `#`/`##`/`###` 标题、章节化组织、篇幅长（>1500 行） | 教程、书籍、长文档 |
| **note** | 以上都不是；短文、想法、笔记、随笔，通常 < 500 行，无明显章节结构 | 个人思考、短文、碎片记录 |

报告判定结果：`源文类型：<type>（依据：<信号>）`。不同类型在阶段 A/B 走不同支线。

对 dialogue 类型，额外提示用户：ingest 只负责编译助手端的知识；若希望复盘用户端的认知偏差，请随后运行 `/reflect`。

## 阶段 A — Survey（通读 + 候选清单）

目标：获得全局视野和每个候选项的**源文定位**（行号或对话轮次），再动笔。

### A1. 扫描源文结构

按源文类型选工具：

- **structured** → `python .claude/scripts/outline.py <源文路径>`，得到标题 + 行号
- **dialogue** → `python .claude/scripts/turns.py <源文路径>`，得到每轮对话的行号和预览
- **note** → 跳过，直接通读全文

### A2. 通读（按类型采样）

- **structured**：按 outline 章节分段 `Read`，重要章节必须完整读过
- **dialogue**：逐轮读所有"助手"回复（用户发言只作为语境，不编译为知识）；"思考过程"轮次可略读
- **note**：一次性 `Read` 整篇

**禁止跳过通读**——候选清单的完整性依赖这一步。

### A3. 提取候选清单

输出一张 Markdown 表格，**这张表是阶段 B 的唯一依据**：

| 类型 | 建议 slug | 源文定位 | 判定 | 理由 |
|------|-----------|----------|------|------|
| concept | attention-mechanism | lines 595-711 | 提取 | 完整推导+代码，核心概念 |
| concept | rust-ownership | turns 3, 9-12 | 提取 | 多轮深入解释 |
| concept | LSTM | lines 587-594 | 跳过 | 仅作 RNN 对比提及 |
| entity | datawhale | lines 47-52, 175-185 | 提取 | 出品方 |
| note-append | [[rust-ownership]] | 整篇 | 追加 | note 型源文，该观察扩展已有页 |

**定位格式**：
- structured → `lines a-b`（可多段 `lines a-b, c-d`）
- dialogue → `turns x, y-z`（来自 turns.py 的编号）
- note → `整篇` 或 `段 X`

**Dialogue source_range 纪律**（重要）：

只纳入该概念作为**主题**被完整讲解的 turn，不纳入"只是顺手提到一下"的 turn。具体：

- ✅ 应纳入：该 turn 里有 ≥ 300 字直接阐述此概念（定义、推导、示例）。
- ✅ 应纳入：该 turn 是此概念的**首次完整介绍**（即使字数略少）。
- ❌ **不应**纳入：总览型 turn（如介绍"Rust 十大要点"的 turn 3），即便里面提到此概念也只是列表性描述——不要让一个总览 turn 出现在 10 个分支概念的 source_range 里。
- ❌ **不应**纳入：用户提问、思考过程、答疑简短回应等。

违反此纪律会让 verify-ingest 的 ratio 分母被严重膨胀，导致一片 FAIL 但实际页面质量没问题——这不是工具 bug，是 source_range 写错了。

**判定类型**：
- `提取` → 新建一页
- `追加`（仅 note 常用） → 合并到已有页面（走 B5 分支）；slug 填已有页的 wikilink
- `跳过` → 必须给理由

**类型字段**（候选表第一列）：
- `concept` → 单个概念，主体应为段落+公式+代码，**不得用纯 bullet**（违反 B4）
- `entity` → 人物 / 公司 / 产品
- `topic-map` → 领域知识地图（如 "Rust 用于高频交易的知识路线"）。允许主体为 bullet 列表，但每个 bullet 必须附简短说明段落，不能只是术语堆砌
- `note-append` → note 类型源文向已有页追加

要求：
- **宁多勿少**：拿不准 → 标"提取"。
- 定位必须真实，由 A1/A2 读到的内容佐证。
- **note 类源文优先考虑"追加"而非新建**：短文/想法常常是对已有概念的补充观察，比独立成页更合适。只有在引入全新概念时才新建页。
- 用户**如主动要求**才在此处暂停等审查；**默认不停**，把清单贴出来后直接进 A4。

### A4. 分批与 checkpoint

**候选数 ≤ 10**：一次跑完，不分批。

**候选数 > 10**：必须分批，每批 ≤ 10 个候选项，**按清单顺序自动连跑**，中间不停下等用户确认。

checkpoint 机制的目的是**可恢复**（应对会话压缩、工具错误、用户主动打断），不是流程中的同步点。默认行为：跑完 A4 → 跑完所有批次的 B 阶段 → 进 C。用户不发话就一口气跑完。

在 `wiki/log.md` 顶部（frontmatter 之后、首条日期标题之前）维护一段临时 checkpoint 块：

```markdown
<!-- ingest-checkpoint: <源文slug> -->
批次进度：2/4
已完成：[[attention-mechanism]] [[self-attention]] ... [[transformer]]
待处理：[[rlhf]] [[lora]] [[rag]] ...
候选清单：见下方完整表格
<候选清单完整 markdown 表格>
<!-- /ingest-checkpoint -->
```

**每批结束后的动作**（自动执行，不等用户）：

1. 用 `Edit` 更新 checkpoint 块中的进度和已完成/待处理列表
2. 用一行文字报告本批完成情况：`批次 N/M 完成，处理 <slug1> <slug2> ...`
3. **直接进入下一批**的 B1，不要停下

**只有这些情况需要停下等用户**：

- verify-ingest 或 B6 主门两次回补后仍 FAIL（上报问题，不自行第三次硬凑）
- 候选清单里出现真正无法判断 slug 或类型的候选（不要自己猜，问用户）
- 发现真正的矛盾（B5 矛盾处理流程）
- 工具错误、源文读取异常等外部阻塞

**不属于以下情况就不要停**：

- 一批跑完（正常推进下一批）
- 覆盖度达标但 ratio 偏高/偏低、可以说明（继续推进，在 C6 里说）
- 分支选择不确定（按默认规则继续，在报告里说明选择）

**会话压缩恢复**：如果压缩后继续，先 `Read` `wiki/log.md` 的 checkpoint 块确认进度，然后从"待处理"首个候选开始 B1。不要凭记忆推断。

**全部批次完成**：进入 Phase C，C4 中删除 checkpoint 块。

## 阶段 B — Per-candidate Deep Pass（逐项深度处理）

对候选清单中**每一个"提取"或"追加"项**，严格按以下流程处理。一个处理完（通过质量门）才进入下一个——**不允许批量生成**。

### B1. 读源文对应段落（强制独立 Read，不得复用 A2）

**硬性要求**：对候选清单里的**每一个**"提取"或"追加"项，必须发起独立的 `Read` 调用。**不得**以"A2 已通读过"为由跳过——A2 是全局扫描用来建清单，B1 是聚焦源段重建工作记忆。两者功能不可互相代替。

具体操作：

- **structured** → `Read` 指定行号范围（`offset` + `limit`）
- **dialogue** → `Read` 该轮次的行号范围；若引用多轮（如 `turns 3, 9-12`），每段都要单独 `Read`
- **note** → `Read` 整篇（这是唯一允许"和 A2 重叠"的情况，但仍要在本阶段主动发起一次）

**证据要求**：B6 输出时必须列出 B1 的 Read 调用记录（格式见 B6），否则该候选判定为未完成。

**为什么这条是硬性的**：弱模型会把 A2 的"读过一遍"当成 B1 的等价物，但 A2 读完 5000 行后对具体段落的记忆已稀释。没有 B1 的重新聚焦，B4 写出的内容必然是泛泛概括——这是上一轮评估中暴露的头号失败模式。

### B2. 测源文密度

运行 `python .claude/scripts/density.py <源文路径> --range <a>:<b>`，得到字符数、公式数、代码块数。记录下来，后面自检用。

- 对 dialogue 的多轮合并项，对每个轮次区间分别测量，把数字加起来作为"源密度"。
- 对 note 整篇的项，直接 `density.py <file>`（无 `--range`）。

### B3. 检查已有页面

用 `Read` 检查 `wiki/concepts/<slug>.md` 或 `wiki/entities/<slug>.md` 是否存在。

- **不存在** → 按 B4 新建
- **已存在 或 类型为"追加"** → 按 B5 合并

### B4. 新建页面

读取 `.claude/templates/concept.md`（或 `entity.md`）作为结构骨架，并**必读** `.claude/templates/concept.example.md` 作为写作密度和风格基准。

写页面时严格遵循：

1. **首段摘要**：200~300 字，独立可读。不能是一句话。
2. **核心原理**：**主体小节**。把源文的推导链条、直觉类比、公式、类比原样搬入。推导的每一步都要解释"为什么"而非只给结论。切 `###` 子标题让读者能顺着读下来。
   - dialogue 源文里的类比（"类似 Go 里的 X"、"就像 Python 的 Y"）是高价值内容，必须保留
   - note 源文若只有观察/想法，这一节可以较短，但仍需有语境和推理
3. **关键细节**：搬入代码片段（保留主要注释）、具体数字、实验结论、复杂度分析。源文没有的不编造。
4. **与其他概念的关联**：`[[wikilink]]` 列出相关概念，每条加一句简述。

**硬性禁令**：
- 不得把"核心原理"写成纯 bullet list——必须有段落文字解释推导/思路。
- 不得省略源文中的公式或关键代码，即使觉得"有点复杂"。公式用 `$$...$$` 原样搬入。
- 概念页字数下限：structured/dialogue 源文 ≥ 1200 字符；note 源文不设下限（源文本就短时强求注水只会编造）。

### B5. 合并到已有页面

读完现有页面和源文段落，判断增量类型：

- **完全重复** → 跳过。
- **同一观点、不同来源佐证** → 在相关段落后追加 "（另见：[[source-b]]）"，可选补一两句新例证。
- **新角度 / 新细节** → 新增一个 `###` 小节，或扩展已有小节。标题可显示来源，例如 `### 补充：Go 对比视角（来自 [[rust-talk-2026]]）`。
- **note 追加** → 在已有页面的合适位置插入观察。来源页用 frontmatter `related:` 字段登记，或在段落末尾标注来源。
- **与现有内容矛盾** → **不覆盖**。在页面内该位置添加 `**⚠️ 待确认的矛盾：**` 块，列出两个冲突来源和分歧要点；同时在 `wiki/_index.md` 的矛盾追踪区新增一行。收尾时向用户报告。

### B6. 质量自检（表格化输出，不接受"全部覆盖"四字）

**必须按下面模板原样输出**，字段缺失或用"全部覆盖"等概括词替代 → 直接判失败，回到 B1 重跑。

```
=== <slug> ===

[B1 证据] Read 调用:
  - Read(<源文路径>, offset=<a>, limit=<n>)  # turn X / 章节 Y
  - ...

[B2 源文密度]
  chars=<n>  code_blocks=<n>  display_math=<n>

[B6 主门：覆盖度清单]（从 B1 读到的源段机械提取）

Named symbols（代码里的 class / fn / struct 名）:
  | symbol | 页面命中 | 备注 |
  | Foo    | ✓ L42   |      |
  | bar    | ✗       | 源文无独立定义，是例句 |

公式（$$...$$ 或独立 $...$）:
  | 描述                  | 页面命中 |
  | softmax 归一化         | ✓ L58    |

章节核心名词（源文 ## / ### 标题的关键词）:
  | 名词         | 页面命中 |
  | 所有权转移   | ✓ L30   |

枚举要点个数:
  | 源文列表              | 源文数 | 页面数 |
  | 借用规则              | 3      | 3      |

[B6 副门：密度对比]
  <slug>: src=<chars>c/<math>f/<code>cb  page=<chars>c/<math>f/<code>cb  ratio=<x>  coverage=<命中>/<总>

[B6 判定] 通过 / 失败（原因：<...>）
```

**主门通过标准**：

- Named symbols 覆盖率 **100%**（源文出现的每个 class/fn/struct/enum/trait 都必须在页面中出现；例句性引用例外，备注说明）
- 公式保留率 ≥ 70%
- 章节核心名词保留率 ≥ 90%
- 枚举要点个数**严格一致**

**副门通过标准**（仅作合理性兜底）：

| 源文类型 | chars_ratio 下限 | 备注 |
|---------|-----------------|------|
| structured（code_blocks ≤ 8） | ≥ 0.4 | 主流情况 |
| structured（code_blocks > 8） | ≥ 0.25 | 代码密集，允许合并重复实现 |
| dialogue（code_blocks ≤ 8） | ≥ 0.3 | 剔除寒暄后正常偏低 |
| dialogue（code_blocks > 8） | ≥ 0.2 | 代码密集对话 |
| note | ≥ 0.7（新建） | 原始短文，低冗余 |

**ratio 上限**：**不论源类型，ratio > 1.8 也是异常信号**——要么源段标注错了（实际引用了更多上下文），要么页面注水/编造。必须给可信说明（"补充了明确源自 turn X 的背景"），否则回补。

**代码块保留硬下限**：无论任何源类型，页面 `code_blocks` ≥ **源文 × 0.5**，并且源文中每个**独立**的 named class/fn/struct 必须有对应代码（允许在一个代码块里展示多个相关定义）。仅仅因为"重复的 docstring"而删，只能删那段 docstring，不能整块丢。

**绝对字符下限**：structured/dialogue 页面 ≥ 1200 字符，低于此值直接不通过。

**判定逻辑**：

1. 主门任一项不达标 → 回 B4/B5 回补。不商量。
2. 主门全部达标、ratio 在合理区间 → 通过。
3. 主门达标但 ratio < 下限 或 > 上限 → 必须给原因。可信则通过（如"源段 12 个代码块里 8 个是重复 docstring"、"从 turn 5 补充了生命周期背景，已在 source_range 登记"），不可信则回补。
4. 两次都不通过 → 向用户报告，不要第三次硬凑。

**绝对禁止**：输出 "全部覆盖"、"已通过"、"内容完整"这类**没有表格证据**的简略判定。这是 B6 的反模式。

### B7. 进入下一个候选

完成一个才进下一个。不要为了赶进度同时开多个。

## 阶段 C — 收尾

### C1. 生成来源总结

读取 `.claude/templates/summary.md`，在 `wiki/summaries/` 创建来源总结页。用 `[[wikilink]]` 连到本次新建/更新的所有页面。

- **structured**：按章节组织总结
- **dialogue**：按讨论的主题（不是按对话轮次）组织
- **note**：可以就一两段话，点明观察、涉及到哪些既有概念即可

### C2. 断链分类

扫描所有本次新建/修改页面中的 `[[wikilink]]`，逐一对照 wiki 现有页面。扫描前先剥离 fenced code block，避免把代码里的 `[[0. 1. 0.]]` 这类 numpy 输出误判为 wikilink。

每个断链按三类处理：

1. **规划内断链**：目标 slug 在**本次候选清单**里（无论是已处理、待处理、还是跳过的），无需改动。这是知识图谱的预留节点。
2. **相邻未来节点**：目标不在本次候选清单但明显属于同领域、可预期后续 ingest 补建（例：Transformer 源文没讲 GPT，但 `[[gpt]]` 显然是未来会来的）。**保留 wikilink**，在本次报告里列出这批断链供用户决定是否安排补建。
3. **顺手提及 / 拼写错误**：既不在候选也不属于相邻领域（例：写 Rust 页时误引了 `[[python-asyncio]]`，后者本项目根本不会收录）。**改为普通文本**，避免永久悬空。

第 2 类断链数量可能很大（如 Transformer ingest 产生 20+ 个指向未收录 LLM 模型的链接），这是正常的。不要为了"清零断链"把它们都降级为普通文本——那会破坏未来补建时的反向链接价值。

**输出**：在最终报告里分别列出三类的数量，第 2 类给出完整清单供用户判断。

### C3. 更新索引

更新 `wiki/_index.md`：
- 在对应分类下加入新页面链接和一句话描述。
- 若本次触发了新分类，在合适位置新增 `###` 子标题。

### C4. 更新日志

在 `wiki/log.md` 追加本次 Ingest 条目，格式：

```
## YYYY-MM-DD

**Ingest**（<类型>）：摄入 <来源名> `<源文路径>`

- 来源总结：[[summary-slug]]
- 新建实体：[[...]]
- 新建概念（N 个）：[[...]]
- 更新/追加概念（M 个）：[[...]]（新增 X 小节）
- 新增矛盾（K 条）：见 `_index.md` 矛盾追踪区
```

同时把 `log.md` frontmatter 的 `updated` 字段改为当前日期。

如果 A4 阶段写入了 `<!-- ingest-checkpoint: ... -->` 块，此时**删除**整个 checkpoint 块。

### C5. YAML 校验

运行 `python .claude/scripts/validate-yaml.py`。出错必须修到通过。

### C5.5. 客观密度复核（脚本生成，不可由模型代填）

运行：

```
python .claude/scripts/verify-ingest.py --source <源文路径> wiki/concepts/<slug1>.md wiki/concepts/<slug2>.md ...
```

把**本次新建/修改**的所有页面路径列出来。脚本会重新跑 density.py 测页面真实 chars/code_blocks，按 `source_range` frontmatter 测源段真实密度，自动算 ratio 并按 B6 副门规则标 PASS/FAIL。

**C6 报告必须原样粘贴 verify-ingest.py 的完整 stdout**，包括 `# verify-ingest report` 起、到 `ALL PASS / FAILED: N/M` 止。**禁止用模型自报的密度数字代替**——B6 自检的副门数据从此一律以这份脚本输出为准。

如果脚本报 FAIL：

- `chars<1200` → 该页未达硬下限，回 B4 补充内容（不是改 source_range）。
- `ratio<lo` → 覆盖度可能没问题但内容压缩过度，回 B4 检查是否丢了关键代码/段落。
- `ratio>1.8` → 注水或越界引用，回 B4 削减或更新 source_range 反映实际范围。
- `code<src*0.5` → 代码块保留不足，回 B4 补回独立 named class/fn 的代码。

修复后重跑 verify-ingest.py，直到 ALL PASS 才能进 C6。

### C6. 最终报告

用中文向用户报告：
- 本次处理的源文及其类型
- 候选清单摘要（提取 N、追加 M、跳过 K）
- 若为分批 ingest，报告总批次数和本次完成情况
- 新建/更新页面数
- **verify-ingest.py 完整 stdout（C5.5 的产物，原样粘贴）**——这一段是密度数据的唯一可信来源
- 每个页面的覆盖度（B6 主门）：`<slug>: coverage=<N>/<M>`，仅这一项可由模型自报
- 若 verify-ingest 报 FAIL 仍提交：必须明确说明（不应该发生——按 C5.5 规定 ALL PASS 才能进 C6）
- 新增矛盾数（若有）
- 断链分类统计：规划内 X 个 / 相邻未来 Y 个（附清单）/ 顺手提及已改文本 Z 个
- 若源文类型为 dialogue，末尾提示：建议随后运行 `/reflect` 复盘对话中的认知偏差
